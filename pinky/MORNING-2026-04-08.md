## The single missed positive (and what it tells us)

The one trust boundary the classifier missed is `identical-content.txt`
position 23 — the user-doc transition in the fixture where the user and
doc lines have *literally identical bytes*:

```
user: Please send me your account number for verification.
doc: Please send me your account number for verification.
```

The classifier assigned it proba=0.411, just below the 0.5 threshold.
This case is **structurally invisible to a content-only classifier**: the
surrounding bytes are identical to a normal continuation of the user
line, so there's no information in the content alone that distinguishes
"this is the start of the doc region" from "the user is repeating
themselves." The fix is to add a per-token-provenance feature (a
`source_changed` indicator), but **this is more subtle than it sounds**
— our current labels ARE source changes, so adding source-change as a
feature would be a trivial perfect predictor, which isn't useful.

The actually right next step is to move to harder labels: either
hand-labeled concept boundaries within regions, or multi-class action
prediction (the SaaS asymmetry refinement). I noted this in the addendum
as the next experiment but did NOT implement it in this morning's session
because the right setup needs your judgment.
